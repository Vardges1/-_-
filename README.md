# Менеджер задач (Next.js + PostgreSQL + Prisma)

Полнофункциональное приложение для **управления задачами**: фильтры, состояние в **URL**, REST API и **опциональная** помощь на базе LLM. Код разложен по слоям (обработчики → сервисы → репозитории), валидация через Zod, единый JSON-формат ответов.

> **По заданию:** во многих ТЗ блок с ИИ **необязателен**; здесь он добавлен **поверх** стабильного CRUD. Приложение полностью работает **без** `LLM_API_KEY` — основные сценарии с задачами доступны; маршруты ИИ при отсутствии ключа возвращают **503**.

---

## Реализованные пользовательские сценарии

| Область | Сценарий |
|---------|----------|
| Задачи | Создание, чтение (одна запись + постраничный список), обновление, удаление |
| Фильтры | Полнотекстовый поиск (`q`, PostgreSQL `to_tsvector` / `plainto_tsquery`, конфиг `simple`), статус, приоритет, окно по сроку (`overdue` / `today` / `week` / `none`) |
| UX | Фильтры синхронизированы с **query-параметрами URL**; смена фильтров сбрасывает на **страницу 1**; пагинация при `total > limit` |
| Подзадачи | Отображаются у задачи; **полная замена** подзадач через API после просмотра декомпозиции от ИИ |
| ИИ (опционально) | Подсказка категории и приоритета, декомпозиция в подзадачи (только предложения), сводка нагрузки (метрики на сервере + текст от LLM) |

## Обязательный функционал и блок с ИИ

**Обязательный (без настройки LLM)**

- PostgreSQL + модели Prisma (`Task`, `Subtask`)
- REST API под префиксом `/api/v1` с общими «конвертами» успеха и ошибки
- CRUD задач, фильтрация списка (Prisma + PostgreSQL FTS для `q`), эндпоинт замены подзадач
- Фронтенд: фильтры, список, форма создания/редактирования в **боковой выезжающей панели**, **диалог** подтверждения удаления, пагинация

**ИИ (нужен `LLM_API_KEY`)**

- `POST /api/v1/tasks/:id/ai/category` — только предложение; пользователь применяет через `PATCH`
- `POST /api/v1/tasks/:id/ai/priority` — аналогично
- `POST /api/v1/tasks/:id/ai/decompose` — предложенные заголовки; пользователь правит и затем **заменяет** подзадачи
- `POST /api/v1/ai/workload-summary` — счётчики **просрочено / срок на этой неделе / открытые с высоким приоритетом** считаются **на сервере**; LLM возвращает только текст поля `summary`

**Провайдеры в духе OpenAI:** задайте **`LLM_BASE_URL`** как корень совместимого API (например `https://api.openai.com/v1` или ваш шлюз). **`LLM_MODEL`** — имя модели у этого провайдера.

## Стек

- **Next.js 15** (App Router) + **TypeScript**
- **PostgreSQL** + **Prisma ORM**
- **Tailwind CSS** (в репозитории нет shadcn)
- **Zod** (тела запросов, query-параметры, **JSON от LLM**)
- **Vitest** + **Testing Library** (небольшой набор тестов)

---

## Архитектурные решения

1. **Тонкие route-handlers** — разбор → валидация (Zod) → вызов **сервиса** → `successJson` / `failureFromZod` / `jsonFromCaughtError`.
2. **Без Prisma в обработчиках** — доступ к данным в репозиториях; сервисы оркестрируют сценарии.
3. **ИИ в отдельном слое** — `src/lib/ai/` (клиент, промпты, схемы) + `AiService`; промпты не живут в React и не в файлах маршрутов.
4. **Не доверять сырому выводу LLM** — разбор JSON, проверка Zod, **одна повторная попытка** с уточняющим промптом, затем структурированная ошибка API при невалидном результате.
5. **Сроки как календарные даты (UTC)** во вспомогательном слое UI, чтобы уменьшить ошибки отображения из‑за часовых поясов.

## Обзор API

**Задачи**

| Метод | Путь | Описание |
|--------|------|----------|
| `GET` | `/api/v1/tasks` | Список + фильтры (`q` = FTS по названию/описанию, `status`, `priority`, `due`, `page`, `limit`) |
| `POST` | `/api/v1/tasks` | Создание |
| `GET` | `/api/v1/tasks/:id` | Карточка + подзадачи |
| `PATCH` | `/api/v1/tasks/:id` | Частичное обновление (тело не пустое) |
| `DELETE` | `/api/v1/tasks/:id` | Удаление |
| `POST` | `/api/v1/tasks/:id/subtasks` | Замена всех подзадач `{ "subtasks": [{ "title" }] }` |

**ИИ** (нужен `LLM_API_KEY`)

| Метод | Путь | Описание |
|--------|------|----------|
| `POST` | `/api/v1/tasks/:id/ai/category` | `{ category, confidence, reason }` |
| `POST` | `/api/v1/tasks/:id/ai/priority` | `{ priority, reason }` |
| `POST` | `/api/v1/tasks/:id/ai/decompose` | `{ subtasks: [{ title }] }` |
| `POST` | `/api/v1/ai/workload-summary` | `{ summary, overdueCount, dueThisWeekCount, highPriorityOpenCount }` |

**Формат ответа**

- Успех: `{ "success": true, "data": … }`
- Ошибка: `{ "success": false, "error": { "code", "message", "details"? } }`

## Переменные окружения

Скопируйте `.env.example` в `.env` и при необходимости измените значения.

| Переменная | Обязательно | Назначение |
|------------|-------------|------------|
| `DATABASE_URL` | **Да** | Строка подключения к PostgreSQL |
| `LLM_API_KEY` | Нет | Если пусто или не задано — эндпоинты ИИ отвечают **503** (`SERVICE_UNAVAILABLE`) |
| `LLM_MODEL` | Нет | По умолчанию `gpt-4o-mini` |
| `LLM_BASE_URL` | Нет | Корень OpenAI-совместимого API; по умолчанию `https://api.openai.com/v1` |

## Локальный запуск

### 1. PostgreSQL (Docker)

```bash
docker compose up -d
```

Дождитесь готовности Postgres (`docker compose ps`).

### 2. Установка и миграции

```bash
npm install
npm run prisma:generate
npm run prisma:migrate
```

При первом запуске `prisma:migrate` создаёт историю миграций и применяет схему. Для одноразовой БД можно вместо migrate выполнить `npx prisma db push` (для продакшена не рекомендуется).

### 3. Демо-данные (по желанию)

```bash
npm run prisma:seed
```

Создаются разнообразные задачи: просроченные, на сегодня, на неделю, завершённые, разные приоритеты/категории, часть задач **с подзадачами** — удобно для фильтров и сводки нагрузки.

### 4. Запуск приложения

```bash
npm run dev
```

Откройте [http://localhost:3000](http://localhost:3000).

## Ограничения и компромиссы

- **Полнотекстовый поиск** — для `q` используется конфиг PostgreSQL `simple` (переносимо, без доп. файлов локали; без языкового стемминга — для преимущественно русского текста можно перейти на `russian` в SQL и миграциях).
- **Нет аутентификации** — сценарий «один пользователь / демо».
- **Нет стриминга** — вызовы LLM только запрос/ответ.
- **Нет фоновых задач** — долгий вызов LLM удерживает HTTP-запрос.
- **Список с подзадачами** — чуть тяжелее ответ; зато UI без N+1 запросов.
- **Замена подзадач** — применение декомпозиции **перезаписывает** текущие подзадачи; в интерфейсе это явно указано.
- **Особенности провайдеров** — часть провайдеров игнорирует `response_format: json_object`; Zod и повторный запрос сглаживают большинство случаев.
